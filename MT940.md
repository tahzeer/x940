# MT940 Format Reference

## Overview

SWIFT MT940 is the Customer Statement Message format used by banks worldwide
for electronic account statements. x940 parses these files into structured
data and exports to JSON, CSV, and camt.053 XML.

## File Structure

An MT940 file contains one or more statements. Each statement starts with a
`:20:` tag and ends at the next `:20:` or end of file.

```
:20: Transaction Reference    ← statement boundary
:21: Related Reference (opt)
:25: Account Identification
:28C: Statement Number / Sequence
:60F: Opening Balance          ← F = final, M = intermediate
:61: Statement Line            ← transaction entry (repeated)
:86: Info to Owner             ← paired with :61: (repeated)
:61: Statement Line
:86: Info to Owner
:62F: Closing Balance
:64:  Closing Available (opt)
:65:  Forward Available (opt)
:86:  Standalone Info (opt)    ← not paired with :61:
:20: Next Statement...
```

## Tag Specifications

### :20: Transaction Reference
- Mandatory, 16 characters max
- Marks the start of a new statement

### :21: Related Reference
- Optional, 16 characters max

### :25: Account Identification
- Mandatory, 35 characters max
- Can be IBAN (`EUR8934567890123456`) or BLZ/Konto (`12345678/0009876543`)

### :28C: Statement Number / Sequence
- Mandatory, format: `5n[/5n]`
- `00342/001`: statement number `00342`, sequence `001`
- `00001`: statement number only, no sequence

### :60F: / :60M: Opening Balance
- Mandatory, exactly one per statement
- Format: `[D|C][YYMMDD][currency][amount]`
- `C260601EUR50000,00`: Credit 50000.00 EUR on 2026-06-01
- `60F` = final (first) opening balance
- `60M` = intermediate opening balance

### :61: Statement Line (Transaction)
- Optional, zero or more per statement
- SWIFT format: `6!n[4!n]2a[1!a]15d1!a3!c16x[//16x][34x]`

| Subfield    | Format | Description |
|-------------|--------|-------------|
| Value date  | 6n YYMMDD | Value date of transaction |
| Entry date  | [4n MMDD] | Entry date (optional, year inferred) |
| D/C Mark    | 2a | D, C, RD, or RC |
| Funds Code  | [1a] | Third currency letter (optional) |
| Amount      | 15d | Comma decimal separator |
| Tx Type     | 1a3c | N or F prefix + 3 chars (NTRF, NMSC) |
| Cust Ref    | 16x | Customer reference |
| Bank Ref    | [//16x] | Bank reference (optional) |
| Suppl.      | [34x] | Supplementary details on continuation line |

**Reversal handling**: RD (Reversal Debit) → treated as credit, RC (Reversal
Credit) → treated as debit. Both set `RvslInd=true` in camt.053.

**Continuation lines**: The `[34x]` supplementary details appear on
continuation lines after `:61:` without a tag prefix.

**Examples**:
```
:61:2606010601D1500,00NTRF//INV991             ← basic debit
:61:0910010930DR62,60NCHGcustomer id//bank id   ← with funds code
:61:110701CN50,00NDISNONREF                     ← no entry date, no bank ref
```

### :62F: / :62M: Closing Balance
- Mandatory, same format as :60F:/:60M:

### :64: / :65: Available Balances
- Optional, same format

## Tag 86 Dialect Formats

The `:86:` tag is the most varied across banks. x940 auto-detects four formats
per-transaction.

### SWIFT Structured (`/KEYWORD/VALUE`)

Used internationally and for SEPA transfers.

```
/EREF/INV-2026-991/REMI/MONTHLY RETAINER FEES/NAME/ALPHA DIGITAL CORP/BIC/ALPHDEFFXXX
```

Common keywords: `EREF`, `REMI`, `NAME`, `BIC`, `IBAN`, `PURP`, `CSID`, `OCMT`,
`CHGS`, `EXCH`, `KREF`, `MREF`, `ISDT`, `TYPE`, `CODE`. All are 3-4 uppercase
letters and handled generically by the decoder.

### German GVC (`?DD` format)

Used by German, Austrian, and Swiss banks (ZKA standard).

```
166?00REMITTANCE?20INV-9924?21KREATOR ABSCHNITT 1?3010020030?3188776655?32ACME CORP GMBH
```

The 3-digit prefix is the GVC (Geschaftsvorfallcode). Common codes:
`166` (domestic transfer), `201` (incoming transfer), `005` (standing order),
`051` (direct debit).

### ZKA Subfield Quick Reference

| Code | Field | Common Usage |
|------|-------|-------------|
| 00 | Posting text | Transaction type description |
| 10 | Prima Nota | Internal bank reference |
| 20-29 | Remittance lines 1-10 | Payment purpose (joined with space) |
| 30 | BLZ | Counterparty bank code |
| 31 | Kontonummer | Counterparty account number |
| 32-33 | Name lines 1-2 | Counterparty name |
| 34 | Textschlussel | Return debit notes |
| 60-63 | Additional info | Extra information |

### Angular (`<DD` / `^DD` format)

Used by Polish, Czech, and Nordic banks.

```
010<00PRZELEW PRZYCHODZACY<20FAKTURA 1234/2026<27JOHN DOE SERVICES<30PL22103004   ← Polish (<)
099^00INSATTNING^20INVOICE 555^27NORDIC TRADING AB                              ← Nordic (^)
```

The first segment before the delimiter is the transaction type code (stored as
`tx_code`). Subfield codes follow the same numbering as GVC.

### Unstructured (Fallback)

Used by US banks, legacy systems, and non-bank sources (PayPal, Venmo). Raw
English prose with no structural delimiters.

```
WIRE TRANSFER OUT TO JOHN DOE FOR MARCH INVOICE 44552 VIA NEW YORK CORE BRANCH
```

Stored under the `detail` key. Always preserves the original text.

## Date & Amount Conventions

- **Dates**: YYMMDD format. Century cutoff: `YY < 80 → 20YY`, `YY >= 80 → 19YY`
- **Amounts**: Comma decimal separator (European convention). `50000,00` = 50000.00
- **Entry date inference**: 4-digit MMDD relative to value date's year with
  December-to-January year-boundary handling

## Key Parsing Rules

1. **Multi-line :86: concatenation**: Strip `\n` without injecting space.
   `"THAT\nSHOULD"` becomes `"THATSHOULD"`
2. **:61: continuation lines**: Insert `\n` separator between the main :61:
   value and supplementary continuation lines
3. **Per-transaction auto-detection**: Each `:86:` independently runs through
   the decoder chain. A single file can mix all four dialect formats
4. **Resolver is priority hint**: `resolver="gvc"` creates chain `[GvcDecoder,
   UnstructuredDecoder]`: unmatched transactions fall through, no errors
